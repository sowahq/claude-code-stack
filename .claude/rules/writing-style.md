# Writing Style — messages, comments, docs

Applies to GitHub comments (issues, PRs, reviews), PR descriptions, commit messages and documentation notes intended for the team.

## Rules

- **No emojis.** Ever.
- **Short and simple.** Stick to the essentials; no filler or marketing jargon ('great', 'perfect', 'robust'…). One to three sentences are usually enough.
- **Factual.** State what has been done, why, and what remains to be done. No superlatives.
- **Direct tone**, simple sentences. No hedging or polite phrases.
- Mention discrepancies honestly (e.g. "not in the mock-up", "to be reviewed if…") without burying them.

## Example

Too long:
> The "Reject a request" scenario (acceptance criterion for this issue) is not present in the Figma mock-up. The page contains only the Interventions, Counter-proposal (Step 1/2) and Empty screens: no rejection design has been created. [...]

Good:
> The 'Reject' outcome is not present in the Figma mock-up (only 'Interventions', 'Counter-proposal' and 'Empty' are present). It has therefore been implemented as a modal, in line with the design system: 'Reject' button, optional reason, confirmation. To be reviewed if a mock-up is provided at a later date.
